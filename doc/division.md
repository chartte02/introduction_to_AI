# 小组分工方案

> 小组人数：4 人  
> 确定日期：2026-05-31

---

## 分工总览

| 成员 | 角色 | 占比 | 核心职责 |
|------|------|------|---------|
| **A** | 数据与特征工程 | **22%** | 数据 EDA、文本预处理、DataLoader |
| **B** | 分类模型 | **26%** | BERT 分类器、交叉验证、调参训练 |
| **C** | 可解释性 | **24%** | LLM 解释生成、注意力可视化 |
| **D（组长）** | 项目集成 + 文档 + 报告 | **28%** | 环境搭建、集成脚本、README、报告 |

---

## 成员 A — 数据与特征工程（22%）

### 负责范围

从原始数据到模型能消费的输入。

### 具体任务

| # | 任务 | 产出 |
|---|------|------|
| 1 | 数据探索性分析（EDA） | 文本长度分布、label 分布、event 分布、可视化图表 |
| 2 | 文本预处理 pipeline 实现 | `src/preprocess.py`（5 步清洗：HTML 解码 → URL 替换 → @mention 替换 → # 去除 → 空白合并） |
| 3 | 分词与词汇分析 | 谣言 vs 非谣言的词频差异、关键词提取 |
| 4 | PyTorch 数据加载器 | `src/dataset.py`，支持按 event 分组采样 |
| 5 | 向报告提供数据分析素材 | 数据集特征表格、文本统计图表 |

### 关键交付节点

- 第 1 周：预处理函数完成（B、C 依赖此项）

---

## 成员 B — 分类模型训练与评估（26%）

### 负责范围

把预处理后的文本变成二分类结果（0/1）。

### 具体任务

| # | 任务 | 产出 |
|---|------|------|
| 1 | BERT 分类模型搭建 | `src/model.py`（twitter-roberta-base / bert-base-uncased + 线性分类头） |
| 2 | 留一事件交叉验证框架 | `scripts/cross_val.py`，7 折自动训练 + 评估 |
| 3 | 训练循环与超参调优 | 学习率、batch size、epoch 数、权重衰减 |
| 4 | 最终模型全量训练 | 全部 train.csv 训练，val.csv 评估 |
| 5 | 评估报告生成 | 混淆矩阵、按事件准确率、7 折均值 ± 标准差 |
| 6 | 向报告提供分类结果分析 | Acc / Precision / Recall / F1、按事件 breakdown |

### 关键交付节点

- 第 2 周：模型搭建 + CV 框架完成
- 第 3 周：调参 + 最终训练

---

## 成员 C — 可解释性模块（24%）

### 负责范围

让模型输出判断依据（一段文字），解释"为什么这么判断"。

### 具体任务

| # | 任务 | 产出 |
|---|------|------|
| 1 | SJTU LLM API 封装 | `src/llm_client.py`，对接 claw.sjtu.edu.cn 接口 |
| 2 | Prompt 模板设计 | 构造 prompt 让 LLM 基于原文 + 分类结果输出判断依据 |
| 3 | 备选方案：注意力可视化 | `src/attention_viz.py`，提取 attention weights 做关键词高亮 |
| 4 | 端到端推理脚本 | `scripts/predict.py`：输入 text → 输出 label + 解释 |
| 5 | 解释质量评估 | 抽样 50 条，人工评判可读性、相关性、准确性 |
| 6 | 向报告提供可解释性分析 | 典型 case study、attention 热力图、LLM 解释示例 |

### 关键交付节点

- 第 2 周：LLM 接口打通 + prompt 模板初版
- 第 3 周：可并行推进（与 B 同时），用 mock 数据先测 LLM 调用
- 第 4 周：解释质量评估

---

## 成员 D — 项目集成 + 文档 + 报告（28%，组长）

### 负责范围

把所有人的模块串起来，保证端到端可运行、可复现、可交作业。

### 具体任务

| # | 任务 | 产出 |
|---|------|------|
| 1 | 项目环境搭建 | `.venv`、`requirements.txt`、目录结构（`src/` `scripts/` `outputs/`） |
| 2 | 训练入口脚本 | `scripts/train.py`，一键启动训练 |
| 3 | 评估入口脚本 | `scripts/eval.py`，一键评估 val.csv |
| 4 | 集成测试 | 确认训练 → 评估 → 推理全链路可运行 |
| 5 | README.md | 环境配置说明、依赖安装步骤、训练/评估/推理命令、复现方式 |
| 6 | report.pdf | 按模板结构撰写，整合各成员结果，≤2000 字 |
| 7 | 代码仓库管理 | 分支策略、PR review、合并冲突解决、提交前检查 |

### 关键交付节点

- 第 1 周：环境 + 目录结构就绪
- 第 2-3 周：集成脚本 + 集成测试
- 第 4 周：README + 报告撰写

## 时间线与依赖关系

```
         第1周          第2周          第3周          第4周
        ├─────────────┼─────────────┼─────────────┼─────────────┤
A:      预处理────────→ 完善 ──────────────────────────────────→
                       ↑ B 依赖 A 的预处理
B:                    模型搭建──────→ CV + 调参 ──→ 最终训练 ──→
                                     ↑ C 依赖 B 的分类结果和 attention
C:                    LLM接口 ──────→ prompt优化 ──→ 解释评估 ──→
D:      搭环境 ──→ 集成脚本 ──────→ 集成测试 ────→ README+报告 →
        ────────────────────────────────────────────────────────→
```

- **B 等 A**：预处理函数完成后才能开始模型搭建
- **C 等 B**：推理脚本需要分类模型产出（可先用 mock 并行开发）
- **D 贯穿全程**：前期搭架子，中期集成，后期写报告收尾

## 分工与评分标准对应

| 评分项 | 分值 | 对应成员 |
|--------|------|---------|
| 大作业报告叙述清楚 | 30 | **D**（主笔）+ A/B/C（提供素材） |
| 代码可运行、部署说明清楚 | 25 | **D**（集成 / README）+ A（预处理） |
| 分类准确率 | 15 | **B**（核心）+ A（数据质量保障） |
| 检测依据的可解释性 | 15 | **C**（核心）+ B（提供 attention 权重） |
| 小组分工协作情况 | 15 | 全员（Git commit 贡献、PR review 参与度） |

## 占比调整机制

如实际执行中出现以下情况，组长可在提交前进行微调（±2%）：

| 触发条件 | 调整方式 |
|---------|---------|
| B 的模型准确率显著高于预期（>85%） | B +2%，从 A/C 各 -1% |
| C 的解释方案特别出彩（LLM 解释 + 注意力双重方案） | C +2%，从 A/D 各 -1% |
| A 额外实现了增强特征（如情感分析、立场检测等） | A +2%，从 B/D 各 -1% |
| 任一成员在自身任务外贡献了额外代码或文档 | 酌情 +2~3% |

---

## 协作规范

1. **代码提交**：每位成员在自己的功能分支上开发，通过 PR 合并到 master
2. **Commit 规范**：使用中文描述，格式 `模块: 做了什么`
3. **沟通频率**：每周至少一次同步进度，组长负责组织
4. **代码风格**：所有注释和文档使用中文，变量/函数名使用英文
5. **可复现性**：所有随机种子固定为 42，超参记录在配置文件中

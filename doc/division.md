# 小组分工方案

> 小组人数：4 人  
> 确定日期：2026-05-31  
> 修订日期：2026-06-01（调整：预处理由 B 承担，A 转向数据分析与模型评估可视化）

---

## 分工总览

| 成员 | 角色 | 占比 | 核心职责 |
|------|------|------|---------|
| **A** | 数据分析 + 模型评估 + 报告图表 | **22%** | 数据 EDA、可视化、错误分析、评估指标图表、报告 2.3 节 |
| **B** | 预处理 + 分类模型 | **28%** | 文本清洗、DataLoader、BERT 分类器、交叉验证、调参训练 |
| **C** | 可解释性 | **24%** | LLM 解释生成、注意力可视化、端到端推理脚本 |
| **D（组长）** | 项目集成 + 文档 + 报告 | **26%** | 环境搭建、集成脚本、README、报告主笔、仓库管理 |

---

## 成员 A — 数据分析 + 模型评估 + 报告图表（22%）

### 负责范围

从数据中提炼洞察，让报告"有图有真相"。

### 具体任务

| # | 任务 | 产出 |
|---|------|------|
| 1 | 数据探索性分析（EDA） | 文本长度分布、标签分布、event 分布、可视化图表 |
| 2 | 词汇与语言特征分析 | 谣言 vs 非谣言词云、关键词 TF-IDF 排名、惊叹号/问号/URL 占比对比 |
| 3 | 交叉验证结果可视化 | 7 折 Acc/F1 柱状图、均值 ± 标准差汇总表 |
| 4 | 混淆矩阵与按事件 breakdown | 混淆矩阵热力图、每个事件的准确率和 F1 分组柱状图 |
| 5 | 错误案例分析 | 模型判错样本的共性分析（文本长度？特定事件？标签偏差？） |
| 6 | 消融实验 | 有预处理 vs 无预处理对分类效果的影响对比 |
| 7 | 向报告提供数据分析素材 | 所有图表 + 表格 + 数据特征描述文字，撰写报告 2.3 节「检测结果分析」 |

### 关键交付节点

- 第 1 周：EDA + 词汇分析完成（不依赖 B，立刻可做）
- 第 2-3 周：等 B 训练出结果后，产出评估图表和错误分析
- 第 4 周：完成消融实验 + 报告 2.3 节

---

## 成员 B — 预处理 + 分类模型训练与评估（28%）

### 负责范围

从原始文本到二分类结果（0/1），把原来 A 的预处理也接过来，保证输入输出口径统一。

### 具体任务

| # | 任务 | 产出 |
|---|------|------|
| 1 | 文本预处理 pipeline 实现 | `model/preprocessing.py`（5 步清洗：HTML 解码 → URL 替换 → @mention 替换 → # 去除 → 空白合并） |
| 2 | 数据加载器实现 | `model/data.py`（CSV 加载、RumorDataset、留一事件 CV 划分） |
| 3 | BERT 分类模型搭建 | `model/model.py`（twitter-roberta-base / bert-base-uncased + 线性分类头） |
| 4 | 训练循环与超参调优 | `model/trainer.py`（train_epoch / evaluate / 学习率调度） |
| 5 | 留一事件交叉验证框架 | `scripts/train.py --cv`，7 折自动训练 + 评估 |
| 6 | 最终模型全量训练 | 全部 train.csv 训练，val.csv 评估，保存模型权重 |
| 7 | 评估脚本 | `scripts/eval.py`，一键在 val.csv 上报告指标 |

### 关键交付节点

- 第 1 周：预处理 + DataLoader 完成（C 依赖此项）
- 第 2 周：模型搭建 + CV 框架完成
- 第 3 周：调参 + 最终训练 → A 开始拿结果做图表

---

## 成员 C — 可解释性模块（24%）

### 负责范围

让模型输出判断依据（一段文字），解释"为什么这么判断"。

### 具体任务

| # | 任务 | 产出 |
|---|------|------|
| 1 | SJTU LLM API 封装 | `model/llm_client.py`，对接 claw.sjtu.edu.cn 接口 |
| 2 | Prompt 模板设计 | 构造 prompt 让 LLM 基于原文 + 分类结果输出判断依据 |
| 3 | 备选方案：注意力可视化 | `model/attention_viz.py`，提取 attention weights 做关键词高亮 |
| 4 | 端到端推理脚本 | `scripts/predict.py`：输入 text → 输出 label + 解释 |
| 5 | 解释质量评估 | 抽样 50 条，人工评判可读性、相关性、准确性 |
| 6 | 向报告提供可解释性分析 | 典型 case study、attention 热力图、LLM 解释示例，撰写报告 2.4 节 |

### 关键交付节点

- 第 2 周：LLM 接口打通 + prompt 模板初版（可与 B 并行，先用 mock 数据）
- 第 3 周：集成 B 的分类模型到推理脚本
- 第 4 周：解释质量评估 + 报告 2.4 节

---

## 成员 D — 项目集成 + 文档 + 报告（26%，组长）

### 负责范围

把所有人的模块串起来，保证端到端可运行、可复现、可交作业。

### 具体任务

| # | 任务 | 产出 |
|---|------|------|
| 1 | 项目环境搭建 | `.venv`、`requirements.txt`、目录结构（`model/` `scripts/` `outputs/`） |
| 2 | 训练入口脚本 | `scripts/train.py`，一键启动训练 |
| 3 | 评估入口脚本 | `scripts/eval.py`，一键评估 val.csv |
| 4 | 集成测试 | 确认训练 → 评估 → 推理全链路可运行 |
| 5 | README.md | 环境配置说明、依赖安装步骤、训练/评估/推理命令、复现方式 |
| 6 | report.pdf | 按模板结构撰写，整合 A/B/C 的素材，≤2000 字 |
| 7 | 代码仓库管理 | 分支策略、PR review、合并冲突解决、提交前检查 |

### 关键交付节点

- 第 1 周：环境 + 目录结构就绪
- 第 2-3 周：集成脚本 + 集成测试
- 第 4 周：README + 报告整合定稿

## 时间线与依赖关系

```
         第1周          第2周          第3周          第4周
        ├─────────────┼─────────────┼─────────────┼─────────────┤
A:      EDA + 词汇分析 ──────────→ 评估图表 ──→ 错误分析 + 报告 2.3 →


B:      预处理 + DataLoader → 模型搭建 → CV + 调参 → 最终训练 ──→
                               ↑ B 自闭环（预处理在内）
C:                            LLM接口 ──→ prompt优化 ──→ 解释评估 ──→
                               ↑ 可先 mock 并行开发
D:      搭环境 ──→ 集成脚本 ──────→ 集成测试 ────→ README+报告 →
        ────────────────────────────────────────────────────────→
```

- **A 不依赖任何人**：EDA 和词汇分析第 1 周即可开工
- **B 自闭环**：预处理 + 模型 + 训练都在 B 手上，无跨成员依赖
- **C 等 B 的分类模型**：推理脚本需要（可先用 mock 并行开发）
- **A 等 B 的训练结果**：评估图表和错误分析需要模型产出（第 3 周起）
- **D 贯穿全程**：前期搭架子，中期集成，后期写报告收尾

## 占比调整机制

如实际执行中出现以下情况，组长可在提交前进行微调（±2%）：

| 触发条件 | 调整方式 |
|---------|---------|
| B 的模型准确率显著高于预期（>85%） | B +2%，从 A/C 各 -1% |
| C 的解释方案特别出彩（LLM 解释 + 注意力双重方案） | C +2%，从 A/D 各 -1% |
| A 的 EDA 和错误分析直接帮助 B 提升了准确率 | A +2%，从 B/D 各 -1% |
| 任一成员在自身任务外贡献了额外代码或文档 | 酌情 +2~3% |

---

## 协作规范

1. **代码提交**：每位成员在自己的功能分支上开发，通过 PR 合并到 master
2. **Commit 规范**：使用中文描述，格式 `模块: 做了什么`
3. **沟通频率**：每周至少一次同步进度，组长负责组织
4. **代码风格**：所有注释和文档使用中文，变量/函数名使用英文
5. **可复现性**：所有随机种子固定为 42，超参记录在配置文件中

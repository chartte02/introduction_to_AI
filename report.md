## 1. 任务目标

构建可解释的谣言检测系统：输入一条推文，输出二分类标签（0=非谣言，1=谣言）和判断依据文字。要求模型具备跨事件泛化能力，可在合理时间内完成推理。

---

## 2. 具体内容

### 2.1 实施方案

系统采用「BERT 分类器 + LLM 解释」复合架构：

```
输入推文 → 预处理 → BERT 分类器 → 标签 + 置信度
                   ↘ SJTU LLM API → 判断依据文本
```

**分类模型**：基于 `cardiffnlp/twitter-roberta-base`（1.25 亿参数，推文语料预训练）微调，分类头为 `[CLS] → Dropout(0.1) → Linear(768,2)`。预处理含 5 步清洗：HTML 解码 → URL → `[URL]` → @mention → `@USER` → 去除 `#` → 合并空白，保留 URL 和 @mention 的结构信号同时消除噪音。

**训练策略**：针对数据集中存在的类别不平衡和事件间差异，采用以下组合策略——(1) 留一事件交叉验证：7 折，模型仅以 text 为输入，event 不参与训练，迫使学习跨事件语言模式；(2) 类别加权损失：按类别频率逆比例计算权重；(3) 梯度裁剪 `max_grad_norm=1.0`；(4) 分层划分 10% dev 集，以 macro_f1 监控早停（patience=2）；(5) 阈值校准：在 dev 集上扫描 τ ∈ [0.05,0.95]，选择最优阈值替代默认 0.5。

**可解释性**：采用双重方案——LLM 解释生成（调用 SJTU 校内 API，prompt 引导从信息源可靠性、措辞煽动性、事实可验证性、逻辑一致性、语言特征五维度分析）和注意力可视化（提取 BERT attention weights，聚合后输出关键词排序和高亮文本），两者互补覆盖完整可解释性光谱。

### 2.2 核心代码分析

项目分为四个模块：`model/`（预处理、数据集、分类器、训练器、LLM 客户端、prompt 模板、注意力可视化）、`scripts/`（训练/评估/推理/校准/Backbone 对比/Voting/解释质量评估/解释方式对比/忠实度评估等 12 个入口脚本）、`src/`（EDA、词汇分析、CV 分析、消融实验、评估图表生成）、`outputs/`（所有权重与评估产物）。

关键设计点：(1) `RumorClassifier.load/save` 接口统一模型持久化，支持 `return_attentions` 参数以获取注意力权重；(2) `train_final_model()` 集成类别加权、梯度裁剪、dev 监控、早停和最佳权重保存，单次调用即可完成完整训练；(3) `evaluate()` 从 `threshold.json` 读取校准阈值判定，决策边界与概率估计解耦；(4) `LLMClient` 封装指数退避重试（最多 5 次）、4xx/5xx 错误区分处理；(5) `compare_explain_modes.py` 支持注意力/LLM/混合三种解释方式横向对比，`eval_faithfulness.py` 量化解释与模型决策的一致性。

#### 消融实验：预处理的效果

为验证预处理 pipeline 的实际贡献，我们设置了对照实验：在相同训练配置下分别训练「原始推文直接输入」和「经 5 步预处理后输入」两个版本的模型。

| 指标 | 无预处理 | 有预处理 | 变化 |
|------|---------|---------|------|
| Accuracy | 88.03% | **88.78%** | +0.75% |
| Macro F1 | 0.8782 | **0.8831** | +0.0049 |
| 谣言 Precision | 0.8671 | **0.9452** | +0.0781 |
| 谣言 Recall | 0.8571 | 0.7886 | -0.0685 |
| 谣言 F1 | 0.8621 | 0.8598 | -0.0023 |

**分析**：有预处理的模型在准确率（+0.75%）和谣言精确率（+0.078↑，从 86.7% 到 94.5%）上均有提升，这意味着预处理后模型对「判定为谣言」更加谨慎和准确——FP 从 23 降至 8。但同时谣言召回率有所下降（-0.069），FN 从 25 升至 37。考虑到谣言检测场景中误报（将正常推文错判为谣言）的社会代价更高，我们选择高精确率的有预处理版本作为最终模型。

### 2.4 判断依据分析

#### 端到端推理示例

以下展示一条推文经过系统完整推理流程的实际输出：

---

**输入推文**：

> BREAKING: multiple witnesses report seeing armed men near the Capitol building. Police have blocked all streets. STAY SAFE!! #BreakingNews #DC

**步骤 1 — 预处理**：

| 操作 | 输入片段 | 输出 |
|------|---------|------|
| HTML 解码 | `&amp;` → `&`（本例不涉及） | 不变 |
| URL → `[URL]` | `<br>`（本例不涉及） | 不变 |
| @mention → `@USER` | 无 @mention | 不变 |
| 去除 `#` | `#BreakingNews` `#DC` | `BreakingNews` `DC` |
| 空白合并 | 多余空格 | 归为一个空格 |

清洗后文本：
> BREAKING: multiple witnesses report seeing armed men near the Capitol building. Police have blocked all streets. STAY SAFE!! BreakingNews DC

**步骤 2 — 分类器推理**：

模型加载校准阈值 τ=0.46，前向传播得到概率分布 P(非谣言)=0.12, P(谣言)=0.88。P(谣言) > τ，判定为**谣言**，置信度 88%。

**步骤 3 — 注意力关键词提取**：

从 BERT 12 层注意力中跨层平均聚合，[CLS] token 对各词的关注度排序，取 Top-5 关键词（已合并子词并过滤标点）：

| 排名 | 关键词 | 注意力分数 |
|------|--------|-----------|
| 1 | `BREAKING` | 0.892 |
| 2 | `armed` | 0.781 |
| 3 | `witnesses` | 0.743 |
| 4 | `STAY SAFE` | 0.689 |
| 5 | `blocked` | 0.651 |

注意力高亮文本（**加粗**=高关注，*斜体*=中等关注）：

> **BREAKING**: *multiple* **witnesses** *report* seeing **armed** *men near the* **Capitol** *building*. **Police** *have* **blocked** *all* **streets**. **STAY** **SAFE**!! BreakingNews DC

**步骤 4 — LLM 生成判断依据**：

调用 SJTU DeepSeek 模型，构造 prompt 输入系统指令 + 推文原文 + 标签（谣言）+ 置信度（88%），LLM 返回：

> 1. 该推文使用了"BREAKING"作为标题化修辞，全大写"STAY SAFE!!"带有强烈的恐慌渲染，措辞倾向于引发读者情绪反应而非传递信息。2. 推文声称"多名目击者"但未提供具体的人名、时间或可靠新闻源链接，关键声索缺乏可验证细节。3. "armed men near the Capitol"是高度敏感表述，但没有任何官方来源或引用支持该说法。综合判断，这条推文利用不确定性情境制造紧迫感和恐慌情绪，具备典型的谣言传播特征。

以上完整流程展示了系统如何在一次推理中同时输出分类标签、置信度、模型内部关注的决策关键词，以及面向用户的自然语言解释。其中步骤 3（注意力）和步骤 4（LLM 解释）互为补充——注意力揭示模型"看到"了什么，LLM 用自然语言解释"为什么"。

#### 语言特征差异分析

我们从 URL 包含率、@mention 使用率、#hashtag 使用率、感叹号和问号出现率五个维度对比了谣言与非谣言推文的语言特征差异。最显著的差异在于 #hashtag 使用率：非谣言推文中 77.8% 含有 hashtag，而谣言推文中仅 61.3%，差距达 16.5 个百分点。这表明非谣言推文的作者更倾向于使用话题标签参与公共讨论，而谣言推文较少使用结构化标签，内容更偏向自由叙述。URL 和 @mention 使用率在两类推文中差异不大（均 <1%），说明链接分享和引用行为本身并非谣言的强判别信号。

#### LLM 解释方案

通过精心设计的 system prompt 引导 LLM 从信息来源、措辞情绪、事实可验证性、逻辑一致性、语言特征五个维度分析。`eval_explain.py` 支持批量抽样生成解释并输出 CSV 供人工评判。LLM 解释语义丰富、面向最终用户，但依赖外部 API 和网络。

#### 注意力可视化方案

从 BERT 各层提取 attention weights，跨层跨头聚合后基于 [CLS] 对其他 token 的关注度计算词级重要性。合并 BERT 子词（`##ing` 等）、过滤纯标点后输出 Top-K 关键词排序和 Markdown 高亮文本。该方案零额外推理开销，可离线运行，适合作为 API 不可用时的备选方案。

#### 解释方式对比与忠实度评估

为量化不同解释策略的质量，我们实现了三种解释方式的说服力对比实验（`scripts/compare_explain_modes.py`）：(1) 仅注意力关键词，(2) 仅 LLM 自由分析，(3) LLM + 注意力引导。下图展示了三种模式的流程差异：注意力模式给出模型内部关注的词但对用户不够自然；LLM 模式生成通顺的自然语言解释但未利用模型内部信息；注意力引导的 LLM 模式将 Top-K 关键词嵌入 prompt，让 LLM 在模型关注点上展开分析，理论上兼具准确性和可读性。

> 📷 **插入图片**：`outputs/figures/explain_mode_comparison.png`（三种解释方式流水线对比图）

同时提供 LLM 多模型对比脚本（`scripts/compare_llms.py`）支持 DeepSeek、GLM、Qwen 等模型的效果横向比较，以及忠实度评估脚本（`scripts/eval_faithfulness.py`）评估 LLM 生成解释与模型实际决策依据的一致性。

## 3. 工作总结

### 可解释性成果总览

> 📷 **插入图片**：`outputs/figures/explainability_summary.png`（可解释性成果总览图：含最终性能指标、五维度语言特征雷达图、按事件解释可信度评级、三大关键发现）

### 收获

(1) 事件泛化是谣言检测的核心挑战——CV 与 val 之间 23% 的准确率差距说明模型在见过的事件上够用但面对全新谣言模式仍需持续学习；(2) 阈值校准将判定从 argmax 解耦为概率估计+决策边界，在类别不平衡场景下提升约 2%；(3) LLM 解释+注意力可视化的双重方案覆盖了从用户端到开发端的完整可解释性需求。

### 遇到的问题与解决

#### 问题 1：数据集类别不平衡

数据集中谣言与非谣言样本比例约为 4:6，且不同事件的谣言比例差异极大（event 2 全为谣言，event 1 仅 24.8% 为谣言）。使用标准交叉熵损失时，模型倾向于将多数类（非谣言）的样本也判为多数类，导致谣言召回率偏低。

**解决**：计算按类别频率逆比例的权重（balanced 策略），使得少数类样本产生更大的损失贡献。以最终模型训练为例，非谣言权重 0.856、谣言权重 1.203，迫使模型同等关注两类样本。配合以 macro_f1（而非 accuracy）作为早停和模型选择指标，确保模型在两类上均衡优化。

#### 问题 2：训练过程不稳定

训练初期 loss 曲线震荡剧烈，部分 epoch 的 dev 指标大幅波动。分析发现原因有二：一是 batch 内标签分布不均匀导致梯度方向不稳定；二是少数异常样本产生极大梯度，破坏了参数更新的一致性。

**解决**：(1) 添加梯度裁剪 `max_grad_norm=1.0`，将每次反向传播的梯度范数限制在 1.0 以内，防止单个样本主导梯度更新；(2) 使用 AdamW 优化器，其内置的自适应学习率和权重衰减机制比普通 SGD 具有更好的训练稳定性；(3) 增加 warmup（前 10% 步线性提升学习率），避免训练初期参数剧烈波动。

#### 问题 3：难以确定最佳训练轮数

数据集缺少独立验证集——train.csv 用于训练，val.csv 用于最终评估，但没有标准验证集来监控训练过程和决定何时停止。如果使用 val.csv 调参则会信息泄漏，评估结果不再可信。

**解决**：从 train.csv 中按标签分层划分 10%（284 条）作为 dev 集（使用 `sklearn.model_selection.train_test_split` 的 `stratify` 参数保证谣言比例与原始分布一致），剩余 2556 条用于训练。每轮训练后在 dev 上计算 macro_f1，仅保存最佳 epoch 的权重。若连续 patience=2 轮无提升则早停退出。最终模型在 epoch 3 就达到了 dev 最优（macro_f1=0.8908），epoch 5 触发早停，节省了近一半训练时间。

#### 问题 4：HuggingFace 模型下载失败

`twitter-roberta-base` 托管在 HuggingFace Hub（`huggingface.co`），国内网络无法直接访问，导致首次运行时 `AutoModel.from_pretrained()` 和 `AutoTokenizer.from_pretrained()` 持续超时。

**解决**：在 `trainer.py` 和所有脚本中自动设置环境变量 `HF_ENDPOINT=https://hf-mirror.com` 和 `no_proxy="*"`，将模型下载请求路由到 HuggingFace 国内镜像站，且在所有 `import transformers` 之前执行以确保生效。用户无需手动配置代理或修改 hosts 文件。

#### 问题 5：Windows 控制台中文乱码

训练日志中的中文（如"谣言""准确率"）在 Windows 命令行中显示为乱码或问号。原因是 Windows 控制台默认编码为 cp1252（西欧字符集），不支持中文字符。

**解决**：在需要打印中文的脚本中，通过 `io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8')` 将标准输出流重新包装为 UTF-8 编码。该包装在所有其他 import 之后执行，避免被下游库重置。同时对所有 `.py` 和 `.md` 文件统一使用 UTF-8 编码保存。

#### 问题 6：LLM API 调用偶发失败

调用 SJTU API（`models.sjtu.edu.cn`）时偶发超时或返回 5xx 错误。尤其是连续大量请求时，速率限制（每分钟约 10 次）和网络波动导致部分请求失败，影响批量生成解释的效率和可靠性。

**解决**：在 `LLMClient.chat()` 中实现指数退避重试机制——最多重试 5 次，等待时间依次为 2s、4s、8s、16s。同时区分错误类型：HTTP 4xx（非 429）表示请求本身有问题（如 API key 无效），直接抛出错误而非徒劳重试；5xx 和 429 表示服务器或限流问题，通过重试可恢复。此外，`eval_explain.py` 在批量调用时每 10 次请求后插入 8 秒冷却间隔并在每条请求间加 1 秒延迟，进一步降低速率限制触发概率。

## 4. 课程建议

1. 建议在任务说明中明确指出事件泛化的评估意义——train 和 val 包含相同事件，仅看 val 准确率会高估模型的真实泛化能力，CV 才是更诚实的指标。很多小组可能不会主动做 CV。
2. 建议增加一次中期检查环节，帮助各组尽早发现方向性问题（如数据划分策略、评估指标选择等）。
3. LLM API 的每日额度、速率限制和推荐使用方式建议在课程初期明确告知，以免学生实验时频繁被限流。
